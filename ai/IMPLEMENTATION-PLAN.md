# 從 Rule 庫到可執行系統：實作規劃書

> 這份文件回答使用者的3個核心問題：
> 1. 246條規則（可程式化程度不一、彼此有相依關係）打算怎麼變成實際的程式？
> 2. 怎麼確保每一條規則都有被考量到，不會實作到一半漏掉？
> 3. 不能完全程式化的部分，後續怎麼處理？
>
> 本文件是架構設計文件，與 `ai/PLAN.md`（進度/狀態日誌）分開維護。`ai/PLAN.md` 會保留一行連結指向這裡。

## 一、先分類：246條規則的4種型態

不是所有規則都用同一種方式處理。先把規則依「可程式化程度」+「是否為複合規則」交叉分類：

| 型態 | 特徵 | 範例 | 處理方式 |
|---|---|---|---|
| **A. 底層原子規則** | 可程式化=是，信心=高，不依賴其他規則 | 均線計算公式、KD計算公式、頭頭高底底高定義、K線橫盤突破確認 | 直接寫成純函式，單元測試對照書中數字 |
| **B. 複合規則** | 依賴其他規則（`[[連結]]`），本身是組合邏輯 | 9種均線戰法、33種經典型態、7種選股法 | 等底層(A)全部就緒後，寫成「引用底層函式結果」的組合函式 |
| **C. 部分可程式化** | 邏輯可寫，但關鍵數字門檻書中沒給、是工程補充值 | 均線糾結的「靠攏程度」、洗盤規則的確認天數 | 照樣實作，但門檻抽成**具名參數**存在設定檔，標記「工程預設值待校準」，不當成鐵律 |
| **D. 不可程式化** | 質性描述、依賴書本以外的資料（消息面/主力意圖）、或純粹是操作心法 | 「認清你是誰」風控心法、基本面選股6大類清單(無數字)、假突破的「主力出貨意圖」 | 不勉強寫成布林判斷式，轉成**人工檢核清單**或**機率性輔助訊號**（見第四節） |

這個分類本身也要留紀錄——下面會說明怎麼把它變成機器可檢查的清單，而不是只存在我的腦中。

## 二、實作分層架構：依規則的相依關係由下往上蓋

246條規則裡有大量 `[[規則名稱]]` 交叉引用（例如「短多做多成功條件」引用「均線多頭排列」和「頭頭高底底高多頭趨勢判定」）。這代表規則庫本身已經隱含一個**分層依賴圖**，實作順序必須照這個圖由下往上，不能顛倒：

```
Layer 0｜資料原語（indicators/ 純函式，輸入OHLCV，輸出數值/布林）
├─ 均線計算（均線分類：均線計算公式）
├─ 轉折波與頭頭高低點（趨勢判定分類：轉折波定義與取點演算法）
├─ 單根/組合K線型態辨識（K棒型態分類：35條）
├─ 成交量分類（量價分類：成交量分類與倍數門檻定義）
├─ MACD/KD/RSI/乖離/布林通道計算（技術指標分類：計算公式類規則）
└─ 缺口偵測（缺口分類：缺口基本定義與偵測規則）

Layer 1｜結構分析（在Layer 0之上，需要歷史序列做進一步推導）
├─ 切線/軌道線畫法（切線軌道線分類：15條，依賴Layer0的轉折點）
├─ 支撐壓力位彙整（支撐壓力分類：20條，依賴Layer0+切線）
└─ 缺口分級與封口判定（缺口分類其餘規則，依賴缺口偵測+支撐壓力）

Layer 2｜訊號判讀（把Layer 0+1組合成有名字的多空訊號）
├─ 均線排列/戰法、葛蘭碧8大法則（均線分類其餘規則）
├─ 指標背離、鈍化、交叉判讀（技術指標分類其餘規則）
├─ K線組合訊號（機械化交易規則、上升三法等）
└─ 33種經典型態總覽（複合規則，B型態的典型代表）

Layer 3｜策略/決策（結合Layer2訊號 + 帳戶/持股狀態）
├─ 選股策略（選股策略分類：19條，依賴Layer2訊號做AND/OR組合）
├─ 停損停利資金管理（停損停利資金管理分類：6條，依賴部位狀態）
└─ 綜合操作SOP（綜合操作SOP分類：9條，最終操作紀律）
```

**實作順序＝Layer 0 → 1 → 2 → 3**，每一層都要先用單元測試釘死，才往上蓋，不然上層一旦出錯很難排查是哪一條底層規則算錯。這個分層也直接對應到程式碼目錄結構（`indicators/` = Layer0/1、`chips/`+`patterns/` = Layer2、`screener/`+`risk/` = Layer3）。

## 三、怎麼確保「246條規則都有被考量到」——用機器檢查，不是靠記憶

這是使用者問的第2個問題，答案不能是「我會很小心」，要有**可驗證的機制**：

### 3.1 建立規則清單（Rule Manifest）
寫一個腳本掃描 `ai/zhu-rules/**/*.md`，從每個檔案的固定欄位（Rule ID、分類、可程式化、信心、`[[連結]]`）自動抽取，產生一份機器可讀的清單 `ai/zhu-rules/_manifest.json`，欄位包含：
```json
{
  "rule_id": "R-MA-08",
  "file": "均線/均線多頭排列.md",
  "programmable": "是",
  "confidence": "高",
  "depends_on": [],
  "implementation_status": "not_started"   // not_started / implemented / tested / calibrated / deferred
}
```
`depends_on` 從內文的 `[[規則名稱]]` 自動解析（比對到其他規則的「名稱」欄位）。這份清單本身用程式產生，不是手key，避免遺漏或抄錯。

### 3.2 程式碼裡用「規則實作登記」機制回連
每一個實作規則的函式，用裝飾器/註解標明它對應哪個 Rule ID，例如：
```python
@implements_rule("R-MA-08")
def is_bullish_aligned(ma5, ma10, ma20, ma60=None) -> bool:
    ...
```
這樣寫一個小工具就能自動核對「manifest 裡有246+條規則」vs「程式碼裡實際登記了幾條」，兩邊對不上的差異就是清楚列出來的「還沒做」清單，不會被淹沒在241個檔案裡漏掉。

### 3.3 覆蓋率報告當成每次開發的檢查點
每次要收工/合併程式碼前，跑 `scripts/check_rule_coverage.py`，輸出格式類似：
```
總規則數：246
已實作並測試：xx
已實作未測試：xx
刻意不實作（D型態，附理由）：xx
尚未處理：xx  ← 這裡如果不是0，明確列出檔名
```
「刻意不實作」也要登記理由（見第四節），不能是空白就直接被當成「忘記了」。這樣不管過多久回來接續，都能一眼看出進度與缺口，不需要重新回想或整份規則庫重讀一遍。

### 3.4 批次D已經做過一次的「反向覆蓋率核對」要常態化
之前用33種經典型態反查246條規則，抓到5個缺口，這個「用複合規則反查底層規則有沒有齊全」的做法很有效，之後應該變成固定流程：**每次新增/修改規則庫，都跑一次「複合規則引用的名稱是否都能在manifest裡找到對應」的連結完整性檢查**（避免 `[[名稱]]` 打錯字或指向不存在的規則，這種斷link在241個檔案裡人工抓很容易漏）。

## 四、不能完全程式化的部分，具體怎麼處理

這是使用者問的第3個問題。依照第一節的C、D兩型態分開講：

### 4.1 型態C（部分可程式化，門檻是工程補充值）
**不要不寫，但要誠實**：
- 邏輯照樣實作，門檻抽成 `config/params.py` 裡的具名參數（沿用之前系統規劃裡就已經決定的 dataclass 參數模式），並在參數旁的註解寫死來源是「書中未給、工程預設值」還是「書中明確數字」，讓後面維護的人一眼分辨哪些數字可以改、哪些改了就違背原著精神
- 每個訊號輸出時，順帶帶出該規則的「信心」欄位（高/中/低），下游的選股清單、健檢報告顯示訊號時可以標示「⚠️ 此訊號門檻為工程預設值，建議搭配回測校準」，不要讓使用者誤以為所有訊號都是同等可靠的書本鐵律
- **校準路徑**：對於「可回測=是」的型態C規則，等歷史資料回補足夠後，用參數掃描（grid search）方式回測不同門檻對應的勝率/期望值，挑出實證上表現最好的數值取代預設值，並把「信心」等級從中→高，同時保留原始工程預設值的紀錄（不要覆蓋掉，方便之後比較）

### 4.2 型態D（不可程式化，質性/依賴外部資料）
分三種子情況，處理方式不同：

**(a) 純操作心法/紀律類**（例如P01的「認清你是誰」、「養成賺錢的習慣口訣」）
→ 不寫成程式邏輯，而是變成**系統的靜態顯示內容**：儀表板或個股健檢報告固定附上一段「操作紀律提醒」，把這些心法原文（改寫過的，非照抄）當成使用者每次看報告時的提醒卡片，功能上更像是「投資紀律檢查清單」而不是自動判斷。

**(b) 有清單但無數字門檻**（例如基本面選股6大類查核清單、盤中選股設定法9種條件裡7種沒量化）
→ 不要自己編數字去湊，而是：
   1. 系統照樣抓取清單裡提到的原始資料欄位（股本、營收年增率、本益比等），呈現給使用者看，但**不自動篩選**，改成「使用者可自訂門檻」的設定項——這樣使用者可以依照自己的判斷填入門檻，系統幫忙算，而不是我們假造朱老師沒講的數字當成他的規則
   2. 這類規則在健檢報告中標示為「參考資料」而非「訊號」

**(c) 依賴主力意圖／消息面等主觀判斷**（例如假突破vs洗盤的意圖判斷、「調節量/換手量/出貨量」三種高檔爆量的區分）
→ 這正是原始企劃的**兩大支柱要合體的地方**：技術面本身無法100%客觀分辨「這是主力出貨的假突破」還是「這是主力洗盤後要噴出」，但**三大法人籌碼資料可以提供額外的機率佐證**——例如同一天出現「疑似假跌破」訊號時，若當天三大法人是net buy（買超），洗盤/進貨的機率就上修；若是net sell，出貨/假突破的機率上修。
→ 具體做法：這類規則的輸出**不是布林值，而是一個「傾向分數」**（例如 -1到+1，或「偏多/中性/偏空」三態），分數來源＝技術面訊號的方向 + 籌碼面買賣超方向的加權綜合，並在報告中清楚寫出「此為機率性輔助判斷，非確定訊號」，避免使用者誤把或然率當成鐵律。

### 4.3 一張表總結型態D的3種子情況與對應輸出形式
| 子情況 | 範例 | 系統輸出形式 |
|---|---|---|
| (a) 操作心法 | 大賺小賠、認清自己 | 靜態提醒卡片，非訊號 |
| (b) 清單無門檻 | 基本面6大類、盤中選股9條件 | 顯示原始數據，使用者自訂門檻 |
| (c) 主觀意圖判斷 | 假突破vs洗盤、爆量三分類 | 技術面+法人籌碼的機率傾向分數，非布林訊號 |

## 五、下一步具體行動（等使用者確認後執行）

1. 寫 `scripts/build_rule_manifest.py`：掃描246個規則檔案，產生 `_manifest.json`，並輸出一份「依賴關係圖」（純文字/表格即可，不用真的畫圖）供人工複查連結是否合理。
2. 依第二節的Layer 0開始，選幾條信心最高、最基礎的規則（均線計算、轉折波定義、K線基本型態）先實作+單元測試，驗證「規則→程式碼」這條轉譯流程本身順不順，再決定要不要跑批次化大量轉譯剩下的規則。
3. 沿用先前系統規劃討論過的技術棧（Python + TWSE/TPEx開放資料 + GitHub Actions + Streamlit + LINE/Email），把這份實作規劃書當成程式碼結構的依據，正式回頭接續之前暫停的系統實作規劃。
