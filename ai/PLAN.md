## Context（為什麼做這個專案）

目標：打造一套台股分析系統，以朱家泓老師的技術分析（均線排列、盤整偵測、支撐壓力箱型、突破/假突破、葛蘭碧八法、K棒型態）為主軸，搭配三大法人籌碼與融資融券資料做每日選股，並將結果主動推播給使用者。

專案目錄下 `ref-project/tw_stock_analyzer/` 是使用者既有的一個成熟 PySide6 桌面應用，**僅作寫法參考、不會被引用、複製或修改**，用來借鏡其資料抓取手法與 SAR 計算邏輯；新專案在 `d:\tw_stock` 根目錄從零建置，是完全獨立的程式碼庫。

使用者的關鍵限制：**沒有可公開部署的伺服器**，但希望電腦、手機都能隨時查看選股結果，且要能收到 LINE / Email 即時通知。因此初步討論方向是採用「零維運、全免費雲端服務組合」而非傳統伺服器架構：

| 環節 | 討論中的方案 | 原因 |
|---|---|---|
| 排程抓資料＋跑選股 | GitHub Actions（cron 排程） | 免費、不依賴使用者電腦是否開機 |
| 資料儲存 | SQLite，由 Action 執行後 commit 回 repo | 不需額外資料庫服務，MVP 階段最低成本 |
| 儀表板（手機/電腦皆可看） | Streamlit Community Cloud，限制僅邀請的檢視者可開啟 | 免費、零維運，給一個網址即可跨裝置查看 |
| 即時通知 | LINE Messaging API（個人 Bot 推播給自己）＋ Email（Gmail SMTP） | 使用者指定的兩個管道；LINE Notify 已停用，改用 Messaging API |

⚠️ **以上呈現方式（GitHub Actions + Streamlit Cloud + LINE/Email）是 AI 提出的推薦方案並經使用者初步同意，但使用者在最後確認階段表示還有其他想法要補充，因此這部分視為「討論中」而非定案**，下次對話開始時應該先跟使用者確認是否仍要採用這個方向，或有新的想法。

## 已知風險與待確認事項

1. **TPEx（櫃買中心）open API 的確切路徑**需在實作時逐一查證（官方文件版面常變動）。
2. **TWSE 三大法人/融資融券資料的更新時間通常晚於股價**（股價 14:30 收盤後即有，但法人與融資融券資料常要到傍晚才齊全），排程時間需納入考量。
3. **法人籌碼資料是盤後 T+1 才會更新**，不是即時的，「每日選股清單」屬性上就是收盤後跑一次，非盤中即時訊號。
4. **SQLite 用 git commit 回 repo 的方式持久化**：作法簡單但每次都是整檔異動，長期 repo 體積會增長；這是 MVP 階段最低成本的做法，之後資料量大可換成 Turso/Supabase 等免費雲端資料庫。
5. **LINE Messaging API 個人 Bot 申請、Streamlit Community Cloud 部署與檢視者權限設定、GitHub Actions Secrets 設定**，這幾步都需要使用者自行用自己的帳號操作（AI 無法代為註冊/登入第三方服務）。
6. 電子書內容（`ebook/`）受著作權保護，**不會**逐字複製書中文字進程式碼或文件；技術規則依朱家泓方法論中廣為人知的核心概念轉譯為可執行邏輯（自行設計實作），完成後需使用者核對規則細節與門檻數字是否符合原意。
7. **`ref-project/` 與 `ebook/` 目前是 git 未追蹤狀態**（`git status` 確認），且 `ref-project/tw_stock_analyzer` 內部有自己獨立的 `.git`。本 repo 已連接 GitHub remote (`github.com/blissjoy/tw_stock`)。為避免不小心把受著作權保護的電子書掃描檔、或使用者另一個獨立專案的原始碼推上 GitHub，**實作第一步務必先建立 `.gitignore` 排除 `ref-project/` 與 `ebook/`**。也建議使用者自行確認 GitHub 上這個 repo 目前是 Private。

## ref-project 研究筆記（純參考，不引用程式碼）

已請 Explore agent 詳細讀過 `ref-project/tw_stock_analyzer/` 的關鍵檔案，重點摘要：

- **股價資料抓取**：用 `yfinance.download()`，批次下載＋向量化 `rolling()` 運算（非 ProcessPoolExecutor）。TW 沒有三大法人/融資融券資料來源，所以新專案籌碼與股價都改用 **TWSE/TPEx 官方開放資料**，不用 yfinance。
- **SAR 計算**：非教科書標準版，有魔改（移除「不得高於前兩日低點」限制、翻轉當日重置 AF 並用 `max/min(hp, high)` 設定、多空判斷用 `<=`）。使用者認可這個版本的做法，新專案要沿用同樣的調整邏輯（重新實作，非複製程式碼）。
- **`zhu_analysis.py`（朱家泓心法模組）**：純函式設計、統一整合入口攤平輸出成 dict。核心演算法：
  - `detect_consolidation`：逐日動態擴張演算法（非固定窗口），依連續天數分級盤整強度。
  - `detect_breakout`：以 5 日均量為基準做量能分級（1.2~1.3倍攻擊量／1.5~2倍攻擊天量／2倍以上爆量），這些數字取自朱家泓課程原文。
  - `classify_granville_signal`：葛蘭碧八法，乖離門檻 15%（課程原文 15~20%，取下緣）。
  - `classify_candlestick_pattern`：先判斷近期高低位階再分辨同型態異義K棒。
  - 程式碼裡明確標註每個數字門檻是「課程原文依據」還是「工程預設值」，新專案應該沿用這個標註習慣。
- **交易日曆**：TWSE `holidaySchedule` API + Big5 轉碼 + 記憶體/檔案兩層快取。14:30 收盤判斷邏輯原意存在但目前實作已被簡化失效，新專案要自己決定正確做法。
- **SQLite 手法**：`ON CONFLICT DO UPDATE` upsert、`ALTER TABLE ADD COLUMN` 包 try/except 做欄位遷移、雙庫 `ATTACH DATABASE` 跨庫 JOIN（新專案暫不需要雙庫，此為單一使用者市場資料庫，非個人持股庫）。
- **參數設計**：`@dataclass` + JSON 序列化 + preset 模板，新專案的 `config/params.py` 可採同樣模式。

1. 逐章精讀整理，存放在 `ai/ebook-summary/`，每條重點都要標註**原書頁碼**（方便回頭查找）

**結論**：兩次測試的勝率（28.8%、33.5%）都不是bug，是趨勢跟隨系統的典型特徵——多數小虧+少數大賺，勝率看起來低但期望值是正的。真正該注意的觀察點反而是「頭頭低」這個出場條件本身勝率偏低(22.1%)且佔出場總數的64%，代表這個「提早示警」機制平均而言是在小虧或打平時就出場，值得未來調校（例如是否該再搭配量能或型態確認才觸發，而非單純頭頭低就出場）。另外，單一策略、單一部位、不考慮多檔股票同時持有的回測設計，本身就會低估真實可執行的年化報酬（真實交易者是多檔股票並行操作），2%~5%的年化數字只反映「這一條規則獨立測試」的結果，不是全書246條規則整合選股後的實戰績效。

## Turso 連線問題排查與修正（2026-07-22）

實際申請Turso帳號、把憑證放進`.env`與GitHub Actions/Streamlit Cloud的secrets後，真正動手跑`seed_turso_from_local.py`才發現一連串規劃階段沒預料到的問題：

1. **`libsql`套件在本機Python 3.14環境裝不起來**：這個套件是PyO3/Rust原生binding，沒有這個Python版本的預編譯wheel，需要Rust+cmake從原始碼編譯，直接失敗。改用純Python實作、實測可正常安裝的舊版**`libsql-client`**（`requirements.txt`已更新並加註原因）。
2. **`libsql-client`預設的websocket(hrana)連線模式對目前Turso伺服器不穩定**：`TURSO_DATABASE_URL`的`libsql://`被client轉譯成`wss://`，實測會隨機出現連線hang住或`aiohttp.client_exceptions.WSServerHandshakeError: 400 Invalid response status`；改成同一個host的`https://`（純HTTP batch API）scheme實測穩定且快很多。已在`turso_client.py`寫成`_force_https_scheme()`固定轉換，不使用`.env`原本設定的scheme。
3. **`libsql_client.ClientSync`的實際API跟原本假設的sqlite3風格差很多**：`.execute()`回傳`ResultSet`（`.rows`/`.columns`，沒有`.fetchone()`/`.fetchall()`/`.description`）、沒有`.commit()`（每個statement即時生效）、沒有`.executemany()`但有`.batch()`可一次網路請求送多筆。重寫了`turso_client.py`的`TursoConnection`轉接器：包一層`_ResultSetCursor`補齊cursor慣用介面，`executemany()`優先用`.batch()`（分500筆一個chunk，避免單次請求過大），都沒有才逐列迴圈。新增2個對**真實`libsql_client`套件**（用本機`file:`模式，不需要網路或Turso帳號）的整合測試，取代原本只對假物件的測試，實際驗證這個轉接器對「真正會用到的那個套件」有效。
4. 修正後`seed_turso_from_local.py`（400天歷史資料）成功背景執行中，進度：stocks 1,854、stock_prices已接近跑完（661,771筆）、institutional_investors進行中、margin_trading尚未開始。

⚠️ 這整段印證了`turso_client.py`當初「把套件細節隔離在單一檔案」的設計是對的——套件從假設的`libsql`換成`libsql-client`、連線scheme從`libsql://`改成`https://`、API從假設的sqlite3風格改成真實的ResultSet風格，全部只需要改這一個檔案，`storage.py`與呼叫端完全沒有異動。
