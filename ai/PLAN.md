## Context（為什麼做這個專案）

目標：打造一套台股分析系統，以朱家泓老師的技術分析（均線排列、盤整偵測、支撐壓力箱型、突破/假突破、葛蘭碧八法、K棒型態）為主軸，搭配三大法人籌碼與融資融券資料做每日選股，並將結果主動推播給使用者。

專案目錄下 `ref-project/tw_stock_analyzer/` 是使用者既有的一個成熟 PySide6 桌面應用，**僅作寫法參考、不會被引用、複製或修改**，用來借鏡其資料抓取手法與 SAR 計算邏輯；新專案在 `d:\tw_stock` 根目錄從零建置，是完全獨立的程式碼庫。

使用者的關鍵限制：**沒有可公開部署的伺服器**，但希望電腦、手機都能隨時查看選股結果，且要能收到 LINE / Email 即時通知。因此初步討論方向是採用「零維運、全免費雲端服務組合」而非傳統伺服器架構：

| 環節 | 討論中的方案 | 原因 |
|---|---|---|
| 排程抓資料＋跑選股 | GitHub Actions（cron 排程） | 免費、不依賴使用者電腦是否開機 |
| 資料儲存 | SQLite，由 Action 執行後 commit 回 repo | 不需額外資料庫服務，MVP 階段最低成本 |
| 儀表板（手機/電腦皆可看） | Streamlit Community Cloud，限制僅邀請的檢視者可開啟 | 免費、零維運，給一個網址即可跨裝置查看 |
| 即時通知 | LINE Messaging API（個人 Bot 推播給自己）＋ Email（Gmail SMTP） | 使用者指定的兩個管道；LINE Notify 已停用，改用 Messaging API |

⚠️ **以上呈現方式（GitHub Actions + Streamlit Cloud + LINE/Email）是 AI 提出的推薦方案並經使用者初步同意，但使用者在最後確認階段表示還有其他想法要補充，因此這部分視為「討論中」而非定案**，下次對話開始時應該先跟使用者確認是否仍要採用這個方向，或有新的想法。

## 已知風險與待確認事項

1. **TPEx（櫃買中心）open API 的確切路徑**需在實作時逐一查證（官方文件版面常變動）。
2. **TWSE 三大法人/融資融券資料的更新時間通常晚於股價**（股價 14:30 收盤後即有，但法人與融資融券資料常要到傍晚才齊全），排程時間需納入考量。
3. **法人籌碼資料是盤後 T+1 才會更新**，不是即時的，「每日選股清單」屬性上就是收盤後跑一次，非盤中即時訊號。
4. **SQLite 用 git commit 回 repo 的方式持久化**：作法簡單但每次都是整檔異動，長期 repo 體積會增長；這是 MVP 階段最低成本的做法，之後資料量大可換成 Turso/Supabase 等免費雲端資料庫。
5. **LINE Messaging API 個人 Bot 申請、Streamlit Community Cloud 部署與檢視者權限設定、GitHub Actions Secrets 設定**，這幾步都需要使用者自行用自己的帳號操作（AI 無法代為註冊/登入第三方服務）。
6. 電子書內容（`ebook/`）受著作權保護，**不會**逐字複製書中文字進程式碼或文件；技術規則依朱家泓方法論中廣為人知的核心概念轉譯為可執行邏輯（自行設計實作），完成後需使用者核對規則細節與門檻數字是否符合原意。
7. **`ref-project/` 與 `ebook/` 目前是 git 未追蹤狀態**（`git status` 確認），且 `ref-project/tw_stock_analyzer` 內部有自己獨立的 `.git`。本 repo 已連接 GitHub remote (`github.com/blissjoy/tw_stock`)。為避免不小心把受著作權保護的電子書掃描檔、或使用者另一個獨立專案的原始碼推上 GitHub，**實作第一步務必先建立 `.gitignore` 排除 `ref-project/` 與 `ebook/`**。也建議使用者自行確認 GitHub 上這個 repo 目前是 Private。

## ref-project 研究筆記（純參考，不引用程式碼）

已請 Explore agent 詳細讀過 `ref-project/tw_stock_analyzer/` 的關鍵檔案，重點摘要：

- **股價資料抓取**：用 `yfinance.download()`，批次下載＋向量化 `rolling()` 運算（非 ProcessPoolExecutor）。TW 沒有三大法人/融資融券資料來源，所以新專案籌碼與股價都改用 **TWSE/TPEx 官方開放資料**，不用 yfinance。
- **SAR 計算**：非教科書標準版，有魔改（移除「不得高於前兩日低點」限制、翻轉當日重置 AF 並用 `max/min(hp, high)` 設定、多空判斷用 `<=`）。使用者認可這個版本的做法，新專案要沿用同樣的調整邏輯（重新實作，非複製程式碼）。
- **`zhu_analysis.py`（朱家泓心法模組）**：純函式設計、統一整合入口攤平輸出成 dict。核心演算法：
  - `detect_consolidation`：逐日動態擴張演算法（非固定窗口），依連續天數分級盤整強度。
  - `detect_breakout`：以 5 日均量為基準做量能分級（1.2~1.3倍攻擊量／1.5~2倍攻擊天量／2倍以上爆量），這些數字取自朱家泓課程原文。
  - `classify_granville_signal`：葛蘭碧八法，乖離門檻 15%（課程原文 15~20%，取下緣）。
  - `classify_candlestick_pattern`：先判斷近期高低位階再分辨同型態異義K棒。
  - 程式碼裡明確標註每個數字門檻是「課程原文依據」還是「工程預設值」，新專案應該沿用這個標註習慣。
- **交易日曆**：TWSE `holidaySchedule` API + Big5 轉碼 + 記憶體/檔案兩層快取。14:30 收盤判斷邏輯原意存在但目前實作已被簡化失效，新專案要自己決定正確做法。
- **SQLite 手法**：`ON CONFLICT DO UPDATE` upsert、`ALTER TABLE ADD COLUMN` 包 try/except 做欄位遷移、雙庫 `ATTACH DATABASE` 跨庫 JOIN（新專案暫不需要雙庫，此為單一使用者市場資料庫，非個人持股庫）。
- **參數設計**：`@dataclass` + JSON 序列化 + preset 模板，新專案的 `config/params.py` 可採同樣模式。

1. 逐章精讀整理，存放在 `ai/ebook-summary/`，每條重點都要標註**原書頁碼**（方便回頭查找）

## Turso 連線問題排查與修正（2026-07-22）

實際申請Turso帳號、把憑證放進`.env`與GitHub Actions/Streamlit Cloud的secrets後，真正動手跑`seed_turso_from_local.py`才發現一連串規劃階段沒預料到的問題：

1. **`libsql`套件在本機Python 3.14環境裝不起來**：這個套件是PyO3/Rust原生binding，沒有這個Python版本的預編譯wheel，需要Rust+cmake從原始碼編譯，直接失敗。改用純Python實作、實測可正常安裝的舊版**`libsql-client`**（`requirements.txt`已更新並加註原因）。
2. **`libsql-client`預設的websocket(hrana)連線模式對目前Turso伺服器不穩定**：`TURSO_DATABASE_URL`的`libsql://`被client轉譯成`wss://`，實測會隨機出現連線hang住或`aiohttp.client_exceptions.WSServerHandshakeError: 400 Invalid response status`；改成同一個host的`https://`（純HTTP batch API）scheme實測穩定且快很多。已在`turso_client.py`寫成`_force_https_scheme()`固定轉換，不使用`.env`原本設定的scheme。
3. **`libsql_client.ClientSync`的實際API跟原本假設的sqlite3風格差很多**：`.execute()`回傳`ResultSet`（`.rows`/`.columns`，沒有`.fetchone()`/`.fetchall()`/`.description`）、沒有`.commit()`（每個statement即時生效）、沒有`.executemany()`但有`.batch()`可一次網路請求送多筆。重寫了`turso_client.py`的`TursoConnection`轉接器：包一層`_ResultSetCursor`補齊cursor慣用介面，`executemany()`優先用`.batch()`（分500筆一個chunk，避免單次請求過大），都沒有才逐列迴圈。新增2個對**真實`libsql_client`套件**（用本機`file:`模式，不需要網路或Turso帳號）的整合測試，取代原本只對假物件的測試，實際驗證這個轉接器對「真正會用到的那個套件」有效。
4. 修正後`seed_turso_from_local.py`（400天歷史資料）成功背景執行中，進度：stocks 1,854、stock_prices已接近跑完（661,771筆）、institutional_investors進行中、margin_trading尚未開始。

## 修正Turso併發crash的漏洞：`executescript()`的「IF NOT EXISTS」文字比對範圍不夠

上一輪修過的併發crash（`KeyError('result')`，見前面章節）在使用者reboot Streamlit Cloud app時又發生了一次，這次原因跟上次不完全一樣：使用者reboot當下，我背景在跑的`daily_pipeline.py --date 20260722 --dry-run`（目標是Turso）仍在執行，兩邊撞在一起，重現了同一個底層套件bug——但這次撞到的是`schema.sql`最開頭的`PRAGMA foreign_keys = ON;`，這句不含"IF NOT EXISTS"文字，所以完全沒被上一輪的修法涵蓋到（上一輪的假設「schema.sql只可能是IF NOT EXISTS的DDL」本身就是錯的，忽略了開頭那句PRAGMA）。

用文字比對陳述式內容來判斷「能不能吞掉這個錯誤」本來就是脆弱的做法——真正該判斷的是「這個KeyError本身是不是那個已知的套件bug訊號」，而不是「這句SQL長什麼樣子」。改用**重試**取代文字比對：`TursoConnection._execute_schema_statement_with_retry()`只要抓到`KeyError`（這正是libsql_client 0.3.1在併發衝突下的錯誤訊號特徵，跟statement內容無關）就重試最多3次、遞增間隔(1s/2s)；3次都失敗才真的往外拋。非`KeyError`的例外（代表真正的SQL錯誤）完全不重試、立即拋出，不會被誤蓋掉。

`tests/test_turso_client.py`原本測「IF NOT EXISTS文字比對」的兩個測試改寫成對應新語意：驗證「前兩次撞到衝突、第三次恢復」會重試到成功、「持續失敗到重試次數用完」最終還是會拋出(不是永遠靜默吞掉)、以及非KeyError例外完全不重試立即拋出。445個測試全過。

⚠️ 這次事件也再次印證README已經寫的警語「同一時間只能有一個行程對Turso寫入」是真實會發生的風險，不是理論上的邊角案例——重試機制能讓「短暫撞在一起」自動恢復，但如果背景寫入行程跑得夠久、恰好每次重試視窗都撞上，還是可能失敗，治本的做法還是避免同時對Turso跑兩個寫入行程。

## 🔍 修正：上面兩節對`KeyError('result')`的根因診斷其實是錯的——真正原因是Turso帳號寫入額度用完，不是併發衝突

使用者reboot Streamlit Cloud app後回報還是同一個crash，且明確要求「先驗證再checkin」。用重試次數(3次)、遞增間隔(最多3秒)都無法恢復——每一次呼叫都100%失敗，這跟「短暫併發衝突」的特徵(應該是偶發、重試後多半能恢復)完全不符，逼出一次真正繞過套件表面行為、直接檢視原始資料的排查。

寫了一支小腳本monkeypatch `libsql_client.http.HttpClient.execute`，在它對回應JSON做`["result"]`存取之前先印出完整原始內容，直接對真實Turso（當下我背景的`daily_pipeline.py --date 20260722`仍在執行）發送請求，實測結果：

```
{'message': 'Operation was blocked: SQL write operations are forbidden (writes are blocked, do you need to upgrade your plan?)', 'code': 'BLOCKED'}
```

真相：Turso伺服器用HTTP 200回傳這個JSON（`libsql_client/http.py`的`_send()`只有在HTTP狀態碼不是2xx時才會辨識出錯誤、拋出`LibsqlError`；200但JSON形狀不是預期的`{"result": ...}`時，`_send()`原封不動把這個dict回傳給呼叫端，呼叫端無條件做`response["result"]`才炸出`KeyError`）——這是**Turso免費方案的寫入額度用完，帳號層級封鎖所有寫入**，跟「多個process同時呼叫ensure_schema()互相卡到」完全無關。前面兩節（「修好Turso併發crash」「Turso schema執行改用重試」）的根因診斷都是錯的：文字比對也好、重試機制也好，都只是針對「併發衝突」這個錯誤假說設計的緩解措施，對「帳號寫入被封鎖」這種**持續性**狀態完全無效——這也解釋了為什麼使用者reboot後還是同樣的錯誤，且新版重試機制在使用者端也100%必然失敗（3次重試全部落空才拋出，只是讓crash多花3秒才出現）。

真正的架構問題：**儀表板（`dashboard/app.py`）明明只需要讀取資料，卻在每次連線時都呼叫`ensure_schema()`這個寫入操作**，導致「寫入被封鎖」這種跟讀取無關的問題，把整個唯讀儀表板也拖垮。修正：
- `dashboard/app.py`的`get_conn()`：`ensure_schema()`改包try/except，失敗時用`st.warning()`顯示警告訊息但不中斷，讓連線物件正常回傳——資料表通常早就建好了，讀取不應該被寫入被拒絕波及；真的是全新空資料庫的話，後續查詢會自然出現"no such table"，這是可接受的降級行為。
- **驗證方式**：這次事件本身就是現成的實測環境——Turso當下確實處於寫入被封鎖的狀態，直接把本機儀表板指向真實Turso（不是本機sqlite）重新整理，實測確認：畫面正確顯示黃色警告訊息、候選清單（2026-07-22，30檔）正常從Turso讀出並顯示，證明修法真的解決了使用者回報的crash，不是又一次沒驗證就checkin的猜測。
- `src/data/turso_client.py`與`tests/test_turso_client.py`的docstring/註解更新，移除「併發衝突」的錯誤敘述，改成準確描述：`KeyError('result')`只代表「回應形狀不是預期的」，可能是真正瞬間性問題(重試有機會恢復)，也可能是像BLOCKED這種持續性狀態(重試無法恢復)；保留重試機制是為了不放棄真正瞬間性的問題，但呼叫端不能假設它一定會成功。README同步修正，加上「檢查Turso Dashboard用量/方案」的實際行動指引，取代原本錯誤的「併發衝突」說法。
- 停止了背景一直在跑、實際上每一次寫入都注定失敗的`daily_pipeline.py --date 20260722`背景工作（continue下去沒有意義，Turso帳號額度沒解決之前任何寫入都會是BLOCKED）。

⚠️ 這次事件的教訓：`KeyError('result')`這種完全不含上下文的裸例外，光靠「重現→猜測→寫防禦性程式碼→測試通過→checkin」的流程是危險的——寫的regression test其實是在驗證「我對bug成因的假設」，如果假設本身是錯的，test再怎麼過都只是驗證了錯誤的心智模型，不是驗證了真正的問題有沒有解決。真正該做的第一步是先看清楚原始錯誤資料（這裡是繞過函式庫、印出raw HTTP response），再決定要修什麼。下一步：使用者需要自行到Turso Dashboard確認帳號用量/方案狀態，額度問題解決（升級方案或等待額度重置）之前，`daily_pipeline.py`等寫入流程會持續失敗（但這是預期中的失敗，不是crash，儀表板讀取功能不受影響）。

## 修正daily_candidates重跑選股會殘留舊結果的問題（2026-07-23）

使用者問「daily_candidates是用來做什麼？有用嗎？」，回答時順便發現上面那個「26筆vs7筆」
的殘留資料現象其實不只是測試遺留物，是一個真實的功能缺陷：`storage.upsert_daily_candidates()`
只用`INSERT...ON CONFLICT(date, stock_id, signal_name) DO UPDATE`，從來沒有「先清掉這個
日期的舊紀錄」這一步。`daily_screener.screen_all_stocks()`每次都是用資料庫現有資料從頭
算出『完整』的候選清單，不是增量疊加——如果同一天重跑選股兩次、兩次選中的股票不完全一樣
(例如補資料前後、或使用者手動按了兩次「立即重新篩選」)，第一次選中但第二次不再符合條件的
股票會永遠留在表裡，讓候選清單顯示過時的結果。使用者確認「同一天可能跑好幾次選股，所以
只能留下最正確的data」，明確要求修正。

修法：
- `src/data/storage.py`新增`delete_daily_candidates_for_date(conn, iso_date)`，只刪除
  指定日期的紀錄，不影響其他日期的歷史候選資料。
- `src/screener/daily_screener.py`的`run_screen_and_store()`在寫入這次算出的候選清單前，
  無條件先呼叫這個刪除函式(即使這次算出0檔候選也要刪，代表『今天正確答案就是沒有候選股』，
  不能因為candidates是空list就跳過清除)。
- 新增4個測試：`tests/test_storage.py`測`delete_daily_candidates_for_date`本身(只刪指定
  日期、無資料可刪時不拋例外)；`tests/test_daily_screener.py`測「兩次選中不同股票、舊的
  被正確清掉」「重跑後0候選、舊紀錄仍被清空」「清除只限定當次日期、不影響其他日期」三種
  情境。464個測試全過。
- 對本機真實`data/tw_stock.db`直接呼叫`run_screen_and_store(conn, iso_date="2026-07-23")`
  驗證：修正前殘留的26筆(19筆測試遺留+7筆正式回補)正確清成7筆，跟pipeline原本印出的候選
  數一致，證明修法在真實資料上有效，不只是測試通過。

## K線圖新增滑鼠十字線＋左上角動態資訊框，仿TradingView（2026-07-23）

使用者要求：價格圖跟成交量圖合併成一張（其實`build_candlestick_figure()`本來就已經是

使用者明確要求「先試做，截圖讓我看是不是要的，再改到正式版」，所以整個過程先在
scratchpad寫prototype腳本（不碰正式程式碼），用Playwright對產生的HTML截圖，反覆修正到
使用者確認滿意後才套用到`src/presentation/chart_data.py`／新增的`desktop/chart_render.py`。

試做過程踩到的坑（都是先截圖才發現，不是憑空預期到的）：
- **Plotly原生spike line的"across"模式對上下堆疊子圖無效**：一開始以為`fig.update_xaxes(
  showspikes=True, spikemode="across")`就能讓垂直線貫穿價格圖+成交量圖，實測截圖發現垂直線
  只會畫在滑鼠所在那一格子圖，不會延伸到另一個。改用`page.evaluate()`直接查DOM裡的
  `.spikeline` SVG元素座標才確認：垂直線的y範圍確實只到該子圖的高度。最終改成在
  `plotly_hover`事件裡用JS呼叫`Plotly.relayout(gd, {shapes:[{yref:'paper', y0:0, y1:1, ...}]})`
  動態畫一條貫穿整張圖紙面高度的線，才達成真正「貫穿兩個子圖」的效果。
- **hover事件的customdata取值bug**：一開始用`evt.points[0]`取資料，但`hovermode="x"`下
  `points[0]`不一定是candlestick/bar這個trace(可能是排序在前面的均線scatter)，導致
  `customdata`是undefined、資訊框看起來完全沒反應。改成迴圈`evt.points`找第一個有
  `customdata`的point才修好。
- **均線/切線的線各自跳出自己的hover標籤**：只把candlestick/bar設成`hoverinfo="none"`
  不夠，其餘scatter類的線(均線/切線/軌道線/支撐壓力)也要設成`hoverinfo="skip"`(完全不
  參與hover偵測)，不然畫面會同時跳出好幾個各自獨立的小標籤跟自訂資訊框搶畫面。
- **Playwright測試腳本第一次滑鼠移動有時候hover事件還沒完全準備好**：debug時發現同一個
  座標(650,250)第一次`mouse.move()`不觸發、但接著再移到別的座標就會觸發——這是測試手法
  要注意的細節(先暖機移動一次再移到目標位置)，不是實作本身的問題。

**架構決定**：這個效果分兩層——
- `src/presentation/chart_data.py`(共用)：加上Plotly原生的x/y軸spike line設定，
  Streamlit/PySide6兩個前端都適用，但Streamlit版垂直線只會在單一子圖內顯示(前述"across"
  跨子圖失效的限制)，仍是比完全沒有十字線好的折衷。
- 新增`desktop/chart_render.py`(桌面版專用)：疊加「JS動態畫貫穿兩子圖的垂直線＋左上角

  Streamlit的`st.plotly_chart()`是用自己的React元件重新渲染Figure的JSON規格，不會執行
  `fig.to_html()`裡注入的`post_script`自訂JS，這是Streamlit渲染架構的限制，不是沒做好。
  `desktop/main_window.py`的`_rerender_chart()`改呼叫`render_chart_html(fig, price_df)`

新增`tests/test_chart_render.py`(5個測試：確認x軸spike關閉/y軸spike開啟、candlestick+bar
的hoverinfo是"none"其餘是"skip"、customdata內容正確對應price_df每一列、HTML內嵌正確的

## 找出並修好TPEx歷史資料缺口的真正根因：不是下載limit，是歷史上的FinMind額度失敗（2026-07-23）

使用者問「4995晶達為什麼沒被選出來」，往下查發現TPEx有681檔天數<60天(見前一節)，
使用者接著要求查ref-project的`inventory.db`確認台股總數、比對缺口、確認是否有下載limit。

**釐清**：`inventory.db`其實是ref-project的「使用者個人持股資料庫」(只有7筆)，不是股票
母清單；改用`scanner_cache.db`的`stocks`表(4269筆，含上市2161/上櫃1761/興櫃347)才是
正確的參考清單。

**比對結果**：ref-project的上市+上櫃4碼代號共1965檔，我們系統`stocks`表全部1965檔都有
(0檔完全缺席)，但521檔(TWSE 62+TPEx 459)缺2024-07-22這天的股價紀錄。

**查code裡有沒有下載limit**：`scripts/backfill_history.py`確實有`--tpex-limit`參數，但
預設值是`None`(不限制)。查`backfill_log.txt`證實當初那次2年歷史回補(`--start 2024-07-22
--end 2026-07-22`)**沒有**加這個參數，本來就打算涵蓋全部~1901檔上櫃股票。

**真正根因**：那次歷史回補用的是FinMind逐股抓，跑到中途撞上FinMind每小時額度上限——
`backfill_log.txt`裡有**1098檔**明確記錄「FinMind API 連續3次請求失敗」，最終只有
681/1584檔成功，其餘股票根本沒補到歷史資料，卻沒有人發現、被誤以為已經回補完成。
這個額度問題後來才在`src/data/finmind_client.py`加上主動節流(`_throttle`)修好，但為時
已晚，沒有回頭補救那次失敗的回補。**不是程式碼裡有寫死的股票數量限制**，是執行期失敗
被誤判成功。

用真實yfinance查詢驗證這是可回補的缺口、不是新股沒資料：`2752`/`4609`/`6236`這幾檔
yfinance查得到2023-01-03以來的完整資料，證實這些股票的資料一直都存在，只是我們從沒
抓到過；相對地`6838`這種查不到2024-07-22之前的資料，才是真正的新股。

**修法**：新增`scripts/backfill_tpex_history_gap.py`（一次性維護腳本），用
`src/data/yfinance_client.py`(沒有FinMind額度限制)重新回補。查資料庫得到精確目標：
796檔上櫃股票的最早日期晚於2024-07-22。依使用者指示「從最近日期開始往前下載」，分成
3個日期區間梯次（2026-01-01~07-24、2025-07-22~2026-01-01、2024-07-22~2025-07-22）
逐步往回補，而不是一次拉滿2年。

**實測結果**：796檔缺口中，**575檔成功回補**到2024-07-22或更早，剩餘221檔裡有60檔是

desktop端開word wrap後能完整發揮多行效果；Streamlit端`\n`會退化成空白分隔的單行，
